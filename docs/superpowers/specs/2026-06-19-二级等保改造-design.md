## 1. 背景与目标

### 1.1 项目背景

forge-admin 是一个基于 RBAC 的企业级后台管理系统，当前已具备基础安全能力（JWT 认证、Refresh Token、RBAC、数据权限框架、操作日志、登录日志、Redis 令牌桶限流），但与 **GB/T 22239-2019《信息安全技术 网络安全等级保护基本要求》二级** 仍存在 27 项差距。

### 1.2 改造目标

- **核心目标**：使系统具备通过有资质测评机构出具的"信息安全等级保护二级测评报告（基本符合及以上）"的技术条件。
- **可衡量结果**：所有 P0、P1 技术项闭环；技术类配套文档齐备；自评估报告通过内部预审；具备测评送审条件。

### 1.3 范围边界

| 类型 | 是否覆盖 | 说明 |
|------|----------|------|
| 应用层代码改造（认证、审计、加密、防护、访问控制） | ✅ 覆盖 | 本次主要工作 |
| 配置层改造（生产 profile、密钥加密、Swagger 禁用、CORS 收紧） | ✅ 覆盖 | |
| 部署相关技术规范（HTTPS、备份、网络隔离、主机加固） | ✅ 覆盖（仅文档） | 部署架构未定型，输出规范文档，落地待架构定稿 |
| 技术类配套文档（自评估报告、密码策略规范、审计规范、备份策略） | ✅ 覆盖 | |
| **管理制度文档**（人员、机构、制度汇编、应急预案的"组织层面"内容） | ❌ 不覆盖 | 公司层面推进，需专门咨询公司协助 |
| **物理环境**（机房、门禁、监控） | ❌ 不覆盖 | 取决于最终部署形态 |
| 测评机构对接、定级备案、专家评审 | ❌ 不覆盖 | 公司层面推进 |

### 1.4 关键约束

- **部署架构未定型**：基础设施类要求（HTTPS/TLS、备份机制、网络隔离、入侵检测）以"部署安全规范"形式产出，应用代码不直接实现。
- **不破坏现有功能**：所有改造必须保持现有业务流程可用，关键改造（密码策略、敏感字段加密）提供灰度过渡方案。
- **遵循现有模式**：所有代码改造遵循 CLAUDE.md 中的分层模式（controller → service → mapper）、starter 自动配置机制、命名约定。

### 2.5 跨阶段技术决策

为避免阶段间返工，以下"贯穿多阶段"的技术决策提前对齐：

1. **加密方案统一**：对称 AES-256-GCM（敏感字段、配置加密）+ 密钥来源（环境变量优先，阶段 3 接入 KMS）。
2. **密码策略参数化**：所有密码相关阈值（长度、复杂度、有效期、历史次数、锁定时长）走配置，不硬编码。
3. **日志架构统一**：阶段 1 完成敏感字段脱敏，阶段 2 完成保留时长和归档机制——使用同一套 `OperationLogAspect` 切面，不重建。
4. **配置文件分层**：`application.yml`（默认）→ `application-dev.yml`（开发）→ `application-prod.yml`（生产），敏感项走 jasypt 加密 + 环境变量。
5. **验证码方案**：图形验证码（Hutool Captcha）+ Redis 存储（5 分钟过期），登录请求体加 `captchaKey` + `captchaCode`。
6. **日志哈希链**：按业务类型异步串行（每 `businessType` 一个独立 Redis 队列，消费者单线程顺序消费并计算哈希），不阻塞请求路径。
7. **并发登录策略**：`security.login.single-session` 配置项控制，默认 `true`（生产单点登录），可改 `false`。

### 3.1 子模块 1：身份认证强化（6 项）

#### 改造清单

| 项 | 改造内容 | 主要文件 |
|----|---------|---------|
| 登录失败锁定 | Redis 失败计数 + 阶梯锁定（5 次→15 分钟，10 次→30 分钟，15 次→永久需管理员解锁） | `AuthController.java`、新增 `LoginAttemptService` |
| 图形验证码 | Hutool Captcha 生成 + Redis 存储（5 分钟过期）+ 登录请求体加 `captchaKey`/`captchaCode` 字段 + 用后即删 | 新增 `CaptchaController.java`、修改 `LoginRequest.java`、`AuthController.java` |
| 密码复杂度 | DTO 加 `@Pattern` 正则（长度 8-32、大小写+数字+特殊字符各至少 1 个）+ 配置化开关 | `UserPasswordRequest.java`、新增 `PasswordPolicyProperties` |
| 密码有效期 | `SysUser` 加 `password_update_time` 字段 + 登录时校验（默认 90 天，可配置）+ 过期返回特定 code 强制改密 | `SysUser.java`、`AuthController.java`、`sql/init.sql` |
| 密码历史 | 新表 `sys_user_password_history`（id, user_id, password_hash, create_time）+ 改密时校验最近 5 次 + BCrypt 哈希存储 | 新增 `SysUserPasswordHistory` 实体 + Mapper + Service |
| 首次登录 + 默认密码 | `SysUser` 加 `first_login` 字段 + 管理员重置生成随机 12 位密码（含复杂度）+ 重置后 `first_login=true` + 登录后拦截返回特定 code | `SysUser.java`、`SysUserServiceImpl.java:221`（去掉硬编码 `"123456"`） |

#### 关键设计点

- **配置项集中**：所有阈值（锁定次数、锁定时长、有效期、历史次数、复杂度正则、验证码长度/过期）统一收敛到 `PasswordPolicyProperties` 和 `LoginPolicyProperties`，`application.yml` 暴露 `security.password.*` / `security.login.*`。
- **登录响应扩展**：增加 `LoginResult` 字段 `passwordExpired` / `firstLogin` / `accountLocked`，前端识别后跳改密页或提示。
- **审计联动**：密码修改、锁定、解锁、重置全部接入 `@OperationLog`，业务类型为 `SECURITY`。

### 3.2 子模块 2：数据保护（3 项）

#### 改造清单

| 项 | 改造内容 | 主要文件 |
|----|---------|---------|
| 敏感字段加密 | AES-256-GCM 加密手机号、邮箱、身份证 + MyBatis TypeHandler 自动加解密 + `@EncryptField` 注解标识 | 新增 `EncryptField.java`、`EncryptTypeHandler.java`、`CryptoUtils.java`；修改 `SysUser.java`、`sql/init.sql` |
| 配置加密（jasypt） | 引入 `jasypt-spring-boot-starter` + 数据库密码、Redis 密码、JWT 密钥、微信密钥用 `ENC(...)` 包裹 + 主密钥来自环境变量 `JASYPT_ENCRYPTOR_PASSWORD` + **明确指定算法 `PBEWithHMACSHA512AndAES_256`**（避免 jasypt 默认的弱算法 `PBEWithMD5AndDES`）+ IV 使用随机生成 | `forge-dependencies/pom.xml`、`application.yml`、`application-prod.yml` |
| Redis 启用密码 | 生产 profile 配置 Redis `requirepass` + 现有 Redis 连接增加 `password`（来自 jasypt 加密） | `application-prod.yml`、新增 Redis 启动脚本/部署文档 |

#### 关键设计点

- **加密范围明确**：阶段 1 仅加密 `sys_user.phone`、`sys_user.email`（身份证本系统当前无）；其他表（如未来 `app_user`）按需扩展。
- **密钥管理过渡方案**：阶段 1 主密钥来自环境变量（启动时注入）；阶段 3 接入 KMS 后切换。
- **数据迁移**：提供 V{date}__encrypt-sensitive-fields.sql 迁移脚本，对现有明文数据批量加密；幂等判断靠**统一密文前缀 `ENCv1:`**（加密时强制添加，迁移时若已存在前缀则跳过；解密时识别前缀并剥离）。前缀中的 `v1` 为密钥版本号，便于阶段 3 接入 KMS 后做多版本密钥共存。
- **模糊查询兼容**：加密后无法直接 `LIKE` 查询手机号——保留手机号最后 4 位明文（`phone_suffix` 字段）做精确匹配查询，整号查询走全量解密（数据量小可接受）。

### 3.3 子模块 3：安全防护（3 项）

#### 改造清单

| 项 | 改造内容 | 主要文件 |
|----|---------|---------|
| XSS 过滤器 | 新增 `XssFilter` + `XssHttpServletRequestWrapper`，对 `getParameter` / `getParameterValues` / `getInputStream`（JSON body）内容做 HTML 转义（Hutool `HtmlUtil.escape`）+ 白名单路径放行（如富文本字段） | `forge-spring-boot-starter-web` 新增 `xss/` 包，`AutoConfiguration.imports` 注册 |

| 操作日志敏感字段脱敏 | `OperationLogAspect` 增加脱敏 + 新增 `@SensitiveField(type=PASSWORD/PHONE/EMAIL/ID_CARD)` 注解 + 默认强制脱敏字段（`password`/`oldPassword`/`newPassword`/`captchaCode`）+ 序列化时按类型脱敏 | `OperationLogAspect.java`、新增 `SensitiveField.java` + `SensitiveDataSerializer.java` |

#### 关键设计点

- **XSS 白名单**：富文本字段（如通知公告 `notice.content`）走 `@XssIgnore` 注解放行，由前端用 DOMPurify 处理；其余全部转义。
- **CSP 策略**：阶段 1 用宽松策略（`default-src 'self'; script-src 'self' 'unsafe-inline'`），阶段 2 收紧（移除 `'unsafe-inline'`，nonce-based）。
- **脱敏配置化**：脱敏字段列表 + 脱敏规则（保留首尾几位）走配置，避免硬编码。

### 3.7 测试要点（阶段 1）

| 测试项 | 验证方式 |

| 验证码 | 错误/过期/重复使用三种场景全部拦截 |

| 密码历史 | 改密为最近 5 次任一密码被拒 |

| 敏感字段加密 | 数据库直接查看为密文；API 返回明文；按手机号后 4 位查询有效 |
| XSS 过滤 | `<script>alert(1)</script>` 在所有接口被转义为 `&lt;script&gt;...` |
| 安全响应头 | dev：`curl -I` 验证 `X-Frame-Options`、`X-Content-Type-Options`、`Content-Security-Policy`、`Referrer-Policy`、`Permissions-Policy` 存在；prod（HTTPS）：额外验证 `Strict-Transport-Security` 存在 |
| 操作日志脱敏 | 登录/改密请求体在 `sys_operation_log.request_param` 中 `password` 字段为 `******` |
| 文件上传 | `.exe` / `.sh` / 改扩展名的恶意文件全部被拒；超大文件被拒；路径穿越被拒 |

### 4.1 子模块 1：审计增强（3 项）

#### 改造清单

| 项 | 改造内容 | 主要文件 |
|----|---------|---------|
| 日志保留 6 个月 | `application.yml` 文件日志 `max-history` 改 180 天 + DB 表 `sys_operation_log`/`sys_login_log` 定时归档（按月归档到 `_archive` 表，原表保留近 6 个月） | `application.yml`、新增 `LogArchiveJob`（Quartz） |
| 日志归档 + 防篡改 | 写入时计算 SHA-256 哈希链（`prev_hash` + `canonical_json(record)` → `cur_hash`）+ 归档独立表 + 每周完整性校验 | `SysOperationLog` 加 `prev_hash`/`cur_hash` 字段、`OperationLogAspect` 增强、新增 `LogIntegrityVerifier` |
| 关键安全事件全覆盖 | 密码修改、权限变更、角色分配、用户启停、文件上传、登录锁定/解锁、强制下线等全部接入 `@OperationLog` + 业务类型 `SECURITY` | 现有 `ServiceImpl` 补注解（审计清单见下文） |

#### 关键设计点

- **哈希链算法**：`SHA-256(prev_hash || canonical_json(record))`，`prev_hash` 取同业务类型上一条日志的 `cur_hash`。
- **异步串行实现（按业务类型分队列）**：每个 `businessType` 维护一个独立 Redis List 队列（key: `log:queue:{businessType}`）；写入时入队即返回（不阻塞请求）；单线程消费者按 `businessType` 顺序消费、计算哈希、入库。这样：
  - 请求路径无锁等待
  - 按 `businessType` 分队列避免单点热点
  - Redis List 持久化（AOF `appendfsync everysec`）保证消息不丢
  - 启动时回放未消费日志
- **归档策略**：每月 1 号凌晨将 6 个月前的日志迁移到 `sys_operation_log_archive`（结构相同），同时导出到独立日志文件 `/var/log/forge-admin/archive/`（仅追加，文件权限 root:forge 0640）。
- **完整性校验**：每周日凌晨遍历哈希链，发现断裂/跳号写入 `sys_security_alert` 表并触发告警（邮件/WebSocket）。
- **审计清单**（需 `@OperationLog(businessType=SECURITY)` 覆盖）：
  - 密码：修改、重置、首次改密
  - 账户：启用、禁用、删除、解锁
  - 权限：角色新增/修改/删除、用户-角色分配、角色-菜单分配
  - 会话：强制下线、并发登录踢出
  - 文件：上传、删除、配置变更
  - 配置：系统参数修改、文件存储配置变更

### 4.2 子模块 2：访问控制深化（3 项）

#### 改造清单

| 项 | 改造内容 | 主要文件 |
|----|---------|---------|
| 单条 getById 越权检查 | AOP 拦截 `ServiceImpl.getById`：返回结果后用 `@DataPermission` 上下文校验数据归属（部门/本人），不匹配抛 `AccessDeniedException` | 新增 `DataPermissionCheckAspect`（forge-spring-boot-starter-mybatis）、修补 `SysDeptServiceImpl`、`SysFileConfigServiceImpl`、`SysOperationLogServiceImpl` 等缺失位置 |
| 敏感操作二次确认 | 新增 `@RequireConfirm` 注解 + 确认令牌（Redis 5 分钟 OTP）+ 前端二次确认弹窗 + 后端校验 `X-Confirm-Token` 头 | 新增 `RequireConfirm.java`、`ConfirmTokenService`、`RequireConfirmAspect`、前端 `ConfirmDialog` 组件 |
| 并发登录控制 | 同一用户登录时按策略踢掉之前的活跃会话（配置项 `security.login.single-session`，默认 `true` 单点登录，可改 `false` 多点登录） | `AuthController.login`、`LoginUserSessionServiceImpl`、配置 `security.login.single-session` |

#### 关键设计点

- **getById 校验范围**：仅对标记了 `@DataPermission` 的实体类型生效，未标记的（如系统配置）跳过；管理员（`isAdmin()`）跳过。
- **二次确认触发场景**：删除用户/角色/部门、批量删除、角色权限变更、文件存储配置变更、强制下线、密码重置。
- **确认令牌**：操作前先 `GET /confirm/token`，返回 6 位数字 OTP（Redis key: `confirm:{userId}:{otp}`，5 分钟过期）；操作时携带 `X-Confirm-Token` 头；OTP 用后即删。
- **并发登录策略**：
  - `single-session=true`（默认，生产推荐）：新登录踢掉旧会话，复用阶段 1 的强制下线机制。
  - `single-session=false`：保留旧行为，但限制最多 3 个并发会话。

#### 改造清单

| 项 | 改造内容 | 主要文件 |
|----|---------|---------|
| OAuth2 JWK 持久化 | `JWKSource` 改为从数据库表加载（启动时读取 + 定期轮换）；私钥用 jasypt 加密入库；新增密钥轮换任务（默认 90 天） | `AuthorizationServerConfig.java:91-101`、新增 `JwkStoreService`、新增 `sys_oauth_jwk` 表、`JwkRotationJob` |

## 7. 风险与依赖

| 风险/依赖 | 影响 | 缓解措施 |
|-----------|------|----------|
| 部署架构未定型 | HTTPS、网络隔离、主机加固等基础设施项无法在代码闭环 | 输出部署规范文档；部署架构定稿后由运维/云厂商落地 |
| 管理制度文档不在范围 | 测评送审要求管理制度类文档齐全 | 明确指向公司层面推进；阶段 3 自评估报告中预留对接章节 |
| 敏感字段加密涉及数据迁移 | 现有数据迁移失败可能导致数据丢失 | 幂等迁移脚本 + 灰度过渡（双写期）+ 迁移前全量备份 |
| 密码策略变更影响现有用户 | 用户原密码不符合新策略，登录后强制改密可能造成支持压力 | 分批通知 + 改密窗口期 + 兼容历史密码登录（仅触发改密提示，不阻断） |
| 日志哈希链异步串行可能丢日志 | 异常退出时队列中未消费日志丢失 | Redis List 持久化（AOF `appendfsync everysec`）+ 启动时回放未消费日志 |
| KMS 切换风险 | 阶段 3 KMS 故障导致无法解密 | EnvKeyProvider 作为兜底；KMS 异常时自动降级 + 告警 |
| Spring Boot 升级风险 | 升级到 3.3.x 可能引入兼容性问题 | 阶段 3 单独评估；先升级 minor 版本（3.2.x 最新），充分回归测试后再大版本升级 |
